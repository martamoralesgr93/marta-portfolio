No existía un design system previo en la institución. Tampoco había tokens, nomenclatura compartida ni guía de accesibilidad. Este proyecto definió la infraestructura de diseño desde cero.

S
SITUATION · Contexto de negocio

Múltiples entornos digitales,
sin criterio unificado

El ecosistema digital de CEF arrastraba años de deuda técnica y de diseño. Cada módulo — gestión académica, documentación, procesos internos — había crecido de forma independiente, sin un sistema de diseño subyacente. El resultado era un ecosistema donde cada equipo de desarrollo resolvía los mismos problemas de forma diferente, con patrones incompatibles entre sí. Para la institución, esto tenía un coste directo: tiempo de desarrollo duplicado en cada nuevo módulo, discrepancias constantes entre diseño y código, y una experiencia del estudiante fragmentada que generaba soporte innecesario.

Esto generaba una fragmentación crítica en la experiencia del alumno:

  • Carga cognitiva extrema por interfaces densas y sobrepobladas.
  • Patrones de interacción inconsistentes entre diferentes módulos.
  • Bloqueos de accesibilidad graves (contraste, tipografía) inaceptables para una institución educativa.
T
TASK · El reto

Rediseño, unificación y
cumplimiento WCAG

El objetivo no era un simple "lavado de cara" o restyling visual. El mandato estratégico fue auditar la plataforma, establecer estándares de accesibilidad innegociables (WCAG AA) y sentar las bases del primer Sistema de Diseño escalable para todo el entorno EdTech de la institución.

Restricciones reales del proyecto: Algunos colores corporativos de CEF no superaban el ratio de contraste WCAG 4.5:1 — lo que obligaba a redefinir la paleta de marca, una decisión con implicaciones más allá del diseño. El equipo de desarrollo necesitaba resultados visibles a corto plazo, mientras que construir el sistema bien requeria tiempo previo de definición. Gestioné esa tensión priorizando los módulos de mayor impacto en el journey del estudiante (calendario y login) como primera prueba de estrés del sistema.

A
ACTION · Lo que hice, y por qué

Auditoría, Wireframing
y unificación de la UI

1. Auditoría UX y Triage de problemas

Insight: La auditoría heurística reveló que los problemas no eran estéticos — eran sistémicos. Contraste inferior a 3:1, affordance rota en botones primarios, flujos inconsistentes entre módulos. Ningún problema era aislado: todos venían de la ausencia de un sistema compartido. Hipótesis: Si establecemos primero los estándares (tokens, tipografía, accesibilidad) antes que la UI, los componentes resultantes serán correctos por construcción, no por revisión manual. Decisión: Invertir las primeras semanas en auditar y definir el sistema, antes de diseñar ninguna pantalla. Un enfoque que requirió argumentar a dirección por qué no entregaríamos UI visible de forma inmediata.

2. Definición de Estilos y Accesibilidad (WCAG)

Dado que el producto está dirigido al sector educativo, la accesibilidad era un requisito innegociable. Establecí una nueva paleta de colores corporativos optimizada para garantizar un contraste mínimo (Nivel AA) según las pautas WCAG. También redefiní la tipografía para mejorar la legibilidad y estructuré un sistema de componentes básicos estandarizado.

Evaluación de Accesibilidad · WCAG Nivel AA
AA

Cumplimiento de Contraste y Tipografía

La paleta corporativa y el sistema tipográfico fueron recalibrados para superar la relación de contraste 4.5:1. Esto asegura una legibilidad total para el alumnado y cumple con las normativas internacionales de accesibilidad en plataformas EdTech.

3. Arquitectura de Componentes

Traspasé el foco de "diseñar pantallas" a "diseñar sistemas". Empecé estructurando wireframes de baja fidelidad para resolver la jerarquía del layout base (Navegación, Contenedores, Formularios) antes de inyectar la nueva capa visual.

4. UI Kit y Estandarización Visual

Con la estructura validada, construí un UI Kit centralizado en Figma: tokens de diseño, jerarquía tipográfica matemática y componentes interactivos con todos sus estados (Hover, Active, Disabled). El rediseño del Calendario Académico fue la primera prueba de estrés del sistema.

5. Handoff y Escalabilidad

Para evitar discrepancias entre diseño y código, documenté el comportamiento de cada componente. Este mismo sistema permitió iterar a velocidad récord otros flujos críticos del portal, demostrando su viabilidad como única fuente de verdad técnica.

Claridad visual y
escalabilidad técnica

  • Accesibilidad certificada: Superación del ratio 4.5:1 en todos los componentes interactivos (WCAG Nivel AA).

L
LEARNINGS · Aprendizajes

Lo que construir un sistema
desde cero enseña

El design system no es el entregable. La adopción sí.

El entregable real no era un archivo de Figma con componentes bien organizados — era que el equipo de desarrollo lo usara como referencia única. Eso implicó documentar no solo el aspecto de cada componente, sino su comportamiento: estados, variantes, casos límite y guía de uso. Un UI Kit que no se adopta es una librería abandonada. La diferencia entre uno y otro es la documentación y la confianza del equipo en el sistema.

Empezar por estructura antes que por estética cambia el resultado.

Trabajar con wireframes de baja fidelidad antes de inyectar UI evitó un error habitual: resolver problemas de jerarquía y layout con recursos visuales. Al separar ambas capas, las decisiones de estructura fueron más claras y la UI resultante más consistente. Los componentes no nacieron de preferencias estéticas — nacieron de decisiones funcionales ya validadas.

La accesibilidad como restricción es una palanca, no un bloqueo.

Redefinir la paleta corporativa para cumplir WCAG AA fue una decisión que afectaba a la identidad de marca, no solo al diseño. En lugar de presentarla como una imposición técnica, la argumenté como una mejora del sistema: colores más versátiles, mayor legibilidad en todos los contextos y cumplimiento de normativa para una institución educativa. La restricción regulatoria, bien comunicada, se convirtió en argumento a favor del cambio.

Diseñar sistemas es diseñar para otros diseñadores y desarrolladores, no para usuarios finales.

En este proyecto, mis usuarios directos eran el equipo de desarrollo. Eso implicó pensar en nomenclatura coherente, documentación clara y componentes con variantes que redujeran la ambigüedad. Es una lógica diferente a la del diseño de producto para usuarios finales — y más cercana a la de construir infraestructura. Esa perspectiva me sirve directamente para proyectos de escala donde el impacto se mide en velocidad de iteración del equipo, no solo en métricas de usuario.
